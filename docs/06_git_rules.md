# Git運用ルール

## ブランチ運用

- `main` は安定版として扱う
- 機能追加は `feature/機能名` または `codex/作業名` のような作業ブランチで行う
- 修正は `fix/修正内容` を使う
- ドキュメント作成は `docs/内容` を使ってよい
- 作業がまとまったらPull Requestまたは差分確認をしてから `main` に反映する

## コミットメッセージルール

コミットメッセージは、何を変えたかが分かる短い日本語または英語にする。

例:

- `docs: アプリ概要とMVP範囲を追加`
- `docs: AI分析設計を追加`
- `chore: gitignoreとenv exampleを追加`
- `feat: 会計入力画面を追加`
- `fix: 投資損益の計算を修正`

## コミットしてよいもの

- ソースコード
- ドキュメント
- 設計メモ
- 架空のサンプルデータ
- `.env.example`
- README
- テストコード
- 設定テンプレート

## コミットしてはいけないもの

- 実データ入りCSV
- 実データ入りTSV
- SQLiteファイル
- データベースのバックアップ
- APIキー
- `.env`
- `.env.local`
- `.env.development`
- `.env.production`
- 個人情報
- 銀行、証券、カード、取引先に関する実明細
- ログファイル
- ビルド成果物

## .envの扱い

- `.env` と `.env.*` はGit管理しない
- Gitに入れるのは `.env.example` のみ
- `.env.example` には空のキー名だけを書く
- 実キー、実URL、実トークンは絶対に書かない

例:

```env
OPENAI_API_KEY=
STOCK_API_KEY=
```

## SQLiteファイルの扱い

- `*.db`
- `*.sqlite`
- `*.sqlite3`

上記はGit管理しない。

SQLiteには実データが入る可能性が高いため、開発中に作成されたDBファイルはローカルだけで扱う。サンプルDBが必要な場合も、実データを含めず、架空データだけを入れる。

## CSVファイルの扱い

- `*.csv`
- `*.tsv`

上記は原則Git管理しない。

ただし、将来 `data/sample` に架空のサンプルCSVを置く場合は、`.gitignore` の例外設定を使って管理してよい。

## サンプルデータと実データの分離

サンプルデータ:

- 架空の人物名
- 架空の取引先
- 架空の店舗名
- 架空の金額
- 架空の銘柄または一般的な説明用データ

実データ:

- 自分の収入、支出、資産、投資、口座情報
- 実在の取引先や店舗名
- 実際のCSVエクスポート
- 実際のSQLite DB
- 実際にAIへ渡した個人情報入りプロンプト

実データは必ずローカルだけで扱い、GitHubにはコミットしない。
